3. Bug vs Error

The next is about the difference between Bug and Error.

Error vs Defect vs Failure

Error Defect Failure
Also known as Mistake Also known as Bug, Fault  
ISTQB Definition: A human action that produces an incorrect result. ISTQB Definition: An imperfection or deficiency in a work product where it does not meet its requirements or specifications. ISTQB Definition: An event in which a component or system does not perform a required function within specified limits.
An error can lead to a Defect. For example, a programming error can lead to a defect in the code. A Defect can lead to a Failure. For example, a defect in the financial report code can lead to the failure of generation of financial report.  
Not all errors lead to a Defect. For example, a spelling error in code comment will not lead to any defects. Not all defects lead to a Failure. For example, some defects require very specific inputs or preconditions to trigger a failure, which may occur rarely or never.  

ISTQB - International Software Testing Qualifications Board

In summary

  • A human being can make an error (mistake), which produces a defect (fault, bug) in the program code, or in a document.
  • If a defect in code is executed, the system may fail to do what it should do (or do something it shouldn't), causing a failure.
  • Defects in software, systems or documents may result in failures, but not all defects do so.

4. Urgent vs non-urgent requests

The next point is about the difference between Urgent and non-urgent requests. [DOWN]

The user work can't wait = URGENT

About the urgent request… When we think of urgent requests, they are those in which the user's work cannot wait. For example… [DOWN] (Read points)… These are very simple examples to illustrate the idea that user work cannot wait. With this in mind… what would a non-urgent request be? [DOWN]

  • He needs to deliver a scene by tomorrow and today ASAP does not allow him to checkin the scene, or ASAP does not start.
  • He needs to checkout a scene to start his work and ASAP doesn't do it right, or it just doesn't.

The user found some problem but can continue his work

Unlike an urgent request, if the user founds a problem but can continue their work, then it would be a non-urgent request. 6. How to report a bug

The next point is how to report a bug. We are going to separate this into two steps, although if we wanted we could do it in one step. So, first step… [DOWN]

Step 1 - Create a new issue

The first step is to create the issue. For that, we access the project, in this case, Dev-Support. We go to the Issues section (in the left sidebar), and click the New Issue button.

Next, we write the title of the Issue and, in the description, we select the Bug template that will fill in the description with different sections of information for us to complete. And then click Submit issue. With these steps in mind… [DOWN]

  • Access Dev-Support project
    • https://gitlab.bethel.jw.org/avs/dev-support
  • Go to Issues and click the New issue button.
  • Write the issue Title.
  • In the Description field, select the Bug template.
  • Fill in the template information, and click Submit issue.

7. How to scale urgent requests

The next and last point is how to scale urgent requests. About this… [DOWN]

Urgent request = Issues tagged as Critical

We currently consider that an urgent request is equal to an issue labeled as Critical. [DOWN]

Therefore, if you really have an urgent issue you can assign the label Critical. And a tip on this… [DOWN] (Show how to do it) [RIGHT]

  • If you really have an urgent issue use the label critical.

TIP. Every member of the project can "Subscribe" for every label, so you can receive email notifications if some issue has been labeled with the tag that we have subscribed. Very useful for feedback request labels.
